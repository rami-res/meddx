Проблема

Когнітивні упередження вбивають

До 40–70 % діагностичних помилок спричинені когнітивними упередженнями — і жоден навчальний матеріал не може їх усунути лише нагадуванням у тексті. MedDx протидіє кожному упередженню окремим архітектурним механізмом.

Anchoring

Ефект якоря

Перша гіпотеза стає «якорем» — лікар шукає підтвердження, ігноруючи альтернативи.

→ Hypothesis-агент генерує ≥ 5 гіпотез до будь-якого ранжування
Premature Closure

Передчасне закриття

Діагноз ставиться до повного збору даних — «класична» клініка відволікає увагу.

→ Intake gate: PatientCase блокує pipeline до заповнення всіх полів
Confirmation Bias

Упередження підтвердження

Лікар шукає лише докази на користь вже обраного діагнозу, ігноруючи спростування.

→ Evidence-агент: симетричний retrieval — ЗА і ПРОТИ окремо
Search Satisficing

Зупинка на першому правдоподібному

Щойно знайдено «достатньо» обґрунтовану відповідь — пошук припиняється.

→ Root-Cause агент: чи пояснює діагноз усі знахідки?

Оркестрація

LangGraph State Machine

StateGraph + фази

Жорстка послідовність фаз — агент не може «пропустити» крок або порушити порядок.

class Phase(str, Enum):
  INTAKE       = "intake"
  AWAITING_DATA = "awaiting_data"
  HYPOTHESES   = "hypotheses"
  EVIDENCE     = "evidence"
  CHALLENGE    = "challenge"
  ROOT_CAUSE   = "root_cause"
  SYNTHESIS    = "synthesis"
  DONE         = "done"

interrupt() — Socratic step

Synthesis-агент зупиняє граф до того, як LLM побачить відповідь. Студент ранжує першим.

# synthesis.py
student_ranking = interrupt(question)
# --- граф заморожено ---

# UI: Command(resume=ranking)
output = llm.invoke(context)

MemorySaver checkpointer

Стан зберігається між HTTP-запитами. thread_id = UUID сесії, зберігається в Streamlit session_state і MySQL.

  • AWAITING_DATA loop — студент доповнює форму, граф продовжується з тієї самої точки
  • Synthesis interrupt — студент ранжує, після чого resume відновлює вузол
  • Сесія відновлювана після перезавантаження (якщо SqliteSaver/MySQL checkpointer)

Conditional routing

Після Intake граф маршрутизується залежно від повноти PatientCase:

def route_after_intake(state):
  if state.missing_fields:
    return "intake"      # loop
  return "hypothesis"

Агент 1

Intake — Completeness Gate

PatientCase — 8 обов'язкових полів

class PatientCase(BaseModel):
  chief_complaint:           str | None
  history_of_present_illness:str | None
  past_medical_history:      str | None
  medications:               str | None
  family_history:            str | None
  systems_review:            str | None
  risk_factors:              str | None
  available_investigations:  str | None

Поле = "unavailable" рахується як заповнене — студент не може «пропустити», лише явно позначити недоступним.

Completeness gate (код, не LLM)

Перевірка виконується програмно — LLM не може «домовитися» обійти gate:

missing = state.patient_case.missing_fields()
if missing:
  return {
    "phase": Phase.AWAITING_DATA,
    "missing_fields": missing,
    "intake_message": llm.invoke(...)
  }
🛡 Anti-bias: Premature Closure
Pipeline фізично заблокований, поки всі поля не заповнені або позначені «недоступно». LLM не отримує неповний кейс.

AWAITING_DATA loop в UI

  • LLM генерує питання про відсутні поля мовою студента
  • Форма зберігає вже заповнені поля між ітераціями
  • Відсутні поля підсвічуються червоним
  • Граф продовжується з тієї самої сесії (checkpointer)

Мультимовність

Студент пише українською, польською, англійською — будь-якою мовою. LLM відповідає мовою студента. Пошук у базі літератури нормалізується до англійської.

Агент 2

Hypothesis — ≥ 5 неранжованих гіпотез

Структурований вихід

class Hypothesis(BaseModel):
  id:              str   # h1..h5+
  name:            str
  rationale:       str
  organ_system:    str
  is_must_not_miss:bool

3 архітектурні інваріанти (код)

if len(hyps) < 5:
  raise ValueError("≥5 required")

mnm = [h for h in hyps
       if h.is_must_not_miss]
if not mnm:
  raise ValueError("must-not-miss missing")

systems = {h.organ_system for h in hyps}
if len(systems) < 2:
  raise ValueError("≥2 systems required")

LangGraph автоматично повторює виклик при ValueError.

🛡 Anti-bias: Anchoring + Availability
5 гіпотез генеруються одночасно, до будь-якого ранжування. Devil's Advocate отримує лише список імен у алфавітному порядку — без ратіонале і без рейтингу.

Must-not-miss категорія

Агент зобов'язаний включити принаймні один «небезпечний» діагноз, навіть якщо він малоймовірний:

  • Тромбоемболія легеневої артерії при болю в грудях
  • Субарахноїдальний крововилив при головному болі
  • Септичний артрит при болі в суглобі
  • Менінгіт при лихоманці + головному болі

Evaluation

Golden set перевіряє mnm_recall = 100 % — жоден кейс не повинен пропустити небезпечний діагноз.

Агент 3

Evidence — Симетричний RAG retrieval

Симетричний retrieval

Для кожної гіпотези виконуються два незалежних пошуки:

# LLM генерує 2 запити на гіпотезу:
# FOR:     "evidence supporting {name}"
# AGAINST: "evidence refuting {name}"

for hyp in hypotheses:
  supporting = retriever.search(
    hyp.for_query, k=5)
  refuting = retriever.search(
    hyp.against_query, k=5)

Програмна анти-галюцинація

def assert_citations_grounded(used, retrieved):
  retrieved_ids = {
    c.pmid for c in retrieved
  } | {c.doi for c in retrieved}
  for c in used:
    assert c.pmid in retrieved_ids \
        or c.doi in retrieved_ids

LLM не генерує PMID/DOI — цитати беруться програмно з retrieved-контексту.

🛡 Anti-bias: Confirmation Bias
LLM фізично отримує і підтверджуючі, і спростовуючі докази для кожної гіпотези. Неможливо «забути» пошукати контраргументи.

BGE-M3 — чому?

  • Мультилінгвальна (студент пише будь-якою мовою)
  • Dense + Sparse в одній моделі — не потрібен окремий BM25
  • Запускається локально на GPU — нульова вартість embedding
  • 1024-dim dense → пам'ять ~3.2 MB на 1000 чанків

Qdrant Hybrid Search (RRF)

Prefetch dense  (k=20) →┐
Prefetch sparse (k=20) →┤ RRF fusion → top-k
Payload filter:         │
  year ≥ 2010          →┘
  evidence_level ≤ 5

Idempotent upsert за PMID+chunk_index — безпечно запускати повторно.

Джерела (без API-ключа): Europe PMC (PubMed · PMC · BMC · PLOS · Cureus) → auto-fallback NCBI E-utilities при недоступності EBI сервера. 15 клінічних тем × 100–150 статей = ~1500 статей, ~5000–8000 чанків.

Агент 4

Devil's Advocate — Сліпа адверсаріальна критика

blind_view() — key invariant

Агент отримує лише кейс і список імен гіпотез в алфавітному порядку. Без ратіонале, без ранжування, без must-not-miss флагів:

def blind_view(state):
  return {
    "patient_case": state.patient_case,
    "hypotheses": sorted(
      [h.name for h in state.hypotheses]
    ),
    # NO: rationale, ranking, mnm
  }

Ізоляція через проєкцію стану, не через промпт.

Structured output

class ChallengeReport(BaseModel):
  contradictions:           list[str]
  alternative_explanations: dict[str, str]
  discriminating_test:      str
🛡 Anti-bias: Anchoring
Агент атакує гіпотези, не знаючи яка з них «провідна». Він не може підтвердити вже обраний діагноз — він може лише заперечувати.

Що агент шукає

  • Суперечності — чи є симптоми, що суперечать кожній гіпотезі?
  • Альтернативні пояснення — ключовий симптом може мати іншу причину?
  • Discriminating test — яке одне дослідження найкраще розрізняє топ-2 гіпотези?

В UI

Результат відображається у вкладці «👿 Адвокат диявола» — суперечності, альтернативи, ключове дослідження. Студент бачить ще до ранжування.

Агент 5

Root Cause — Погляд на весь організм

Що отримує агент

Повний контекст сесії — все, що накопичилось:

  • Кейс пацієнта (всі поля)
  • Список гіпотез з ратіонале
  • Зведення доказів (кількість ЗА і ПРОТИ по кожній)
  • Звіт Devil's Advocate (суперечності)

Structured output

class RootCauseAssessment(BaseModel):
  all_findings_explained: bool
  unexplained_findings:   list[str]
  candidate_underlying_conditions:
                          list[str]
  comment:                str | None
🛡 Anti-bias: Search Satisficing
Агент перевіряє: чи пояснює найкраща гіпотеза усі знахідки? Якщо ні — це сигнал про можливу системну причину або другий діагноз.

Приклад логіки

Пацієнт з анемією, жовтяницею і ураженням нирок. Гіпотеза «аутоімунна гемолітична анемія» пояснює перші два, але не ниркову функцію.

⚠️ Агент позначить нирковий компонент як «unexplained_finding» і запропонує перевірити системний васкуліт (SLE) як underlying condition.

Synthesis — Сократівський крок + цитати

Послідовність виконання

# 1. Граф зупиняється ДО LLM
question = _socratic_question(state)
student_ranking = interrupt(question)
#    ↑ UI показує форму ранжування

# 2. Після resume:
output = llm.invoke(
  [system, HumanMessage(context)]
)

# 3. Цитати — програмно, не LLM
ranking = _attach_citations(
  output.ranked_hypotheses, state
)
assert_citations_grounded(used, retrieved)

SynthesisResult

class SynthesisResult(BaseModel):
  ranking:          list[RankedHypothesis]
  workup_plan:      list[str]
  evidence_summary: str
  socratic_feedback:str # ← ключове

socratic_feedback — порівняння рейтингу студента з системним + пояснення розбіжностей.

🛡 Ключова педагогічна цінність
Студент ранжує самостійно, без підказки системи. Лише потім система порівнює і пояснює, де і чому міркування розійшлися.

Що бачить студент (UI)

  • 🥇🥈🥉 Ранжований диференційний ряд із заповіданням правдоподібності
  • 📚 Ключові цитати для топ-3 (PMID/DOI, рівень доказовості)
  • 📋 Планований план обстеження (workup)
  • 💬 Сократівський фідбек на їх ранжування
  • 📊 Якість доказової бази (evidence_summary)

MySQL — транзакційні дані

5 моделей SQLAlchemy 2 для студентської активності:

  • User — email, name, locale
  • DiagnosticSession — thread_id, phase, status
  • Case — patient_case JSON, session FK
  • CaseHypothesis — hypothesis_id, rank_final
  • StudentAnswer — ranking JSON, feedback JSON

Cascade / SET NULL стратегія

User ──SET NULL──→ Session
Session ──CASCADE──→ Case
Case    ──CASCADE──→ Hypothesis
Session ──CASCADE──→ StudentAnswer

Видалення користувача анонімізує сесії (не видаляє навчальні дані).

Спостережуваність

Langfuse — трейсинг 6-агентного пайплайну

Що трейситься автоматично

  • Кожен LLM-виклик (вхід, вихід, токени, вартість)
  • Retrieval запити до Qdrant (latency, k, query)
  • Фаза переходу в StateGraph (span per node)
  • interrupt() та resume() як окремі spans

Self-hosted docker-compose

Langfuse v3 стек — 6 сервісів в одному docker compose up -d:

  • langfuse-web (:3000) — UI
  • langfuse-worker — async event processing
  • PostgreSQL — metadata store
  • ClickHouse — аналітика трейсів
  • Redis — черга
  • MinIO — artifact storage

Інтеграція (zero-config)

# llm/factory.py
from langfuse.callback import CallbackHandler

callbacks = []
if settings.langfuse_public_key:
  callbacks = [CallbackHandler()]

llm = init_chat_model(model,
        callbacks=callbacks)

Без ключів — callbacks = [], трейсинг вимкнено. Нічого не ламається.

  • Середня вартість однієї діагностичної сесії ($)

  • Яка модель дає кращий mnm_recall при меншій ціні?
  • Scores: прив'язати golden-set метрики до трейсів

Майбутній розвиток

Roadmap — від курсового до реального продукту

Короткострокові ~3 місяці
Langfuse Evals — автоматичний скоринг трейсів через golden set + score API
Локальний reranker bge-reranker-v2-m3 поверх hybrid retrieval (↑ precision)
Веб-пошук свіжих публікацій (Tavily/SerpAPI) поверх локального Qdrant-індексу
LangGraph streaming → Streamlit st.write_stream() для live вводу токенів
Input/output guard — захист від prompt injection у медичних полях
Середньострокові ~6 місяців
Персональна аналітика — які упередження типові для конкретного студента?
Spaced repetition — повторення кейсів де були помилки (SM-2 алгоритм)
Multi-user auth (FastAPI + JWT) + список сесій в особистому кабінеті
Клінічний куратор — викладач бачить агреговану статистику групи
Розширення corpus: клінічні настанови WHO/UpToDate-open + підручники (open-access)
Довгострокові ~1 рік
Інтеграція з симуляторами (OpenSimMan, Virtual Patient) — динамічний кейс замість форми
Мультиагентна дискусія — два агенти захищають різні гіпотези, студент арбітр
Дослідницький датасет — анонімний export bias-патернів для медичних досліджень
Мобільний додаток (PWA / React Native) — кейс-завдання між парами
Контекстуальна адаптація — рівень складності залежно від року навчання студента
Ключовий принцип розвитку: кожен новий агент або функція повинні підсилювати антибіасові інваріанти — не послаблювати gate completeness, не розкривати ранжування Devil's Advocate, не генерувати цитати без грандинг-перевірки.
